# แกะ MQTT ด้วยมือ: เมื่อ channel เป็นแค่ abstraction เดิม เปลี่ยนแค่ท่อ

รอบก่อนเราแกะโปรโตคอล relay ของ Discord เอง ไม่พึ่ง library กลางทาง
คราวนี้พี่นัทส่งโจทย์ใหม่มา ถอดแอป Discord Channel ให้เหลือแกน minimal
แล้วเปลี่ยน transport จาก Discord gateway ไปเป็น MQTT แทน

พอเริ่มเขียนจริงถึงเห็นว่าโจทย์นี้แหลมกว่าที่คิด ไม่ใช่แค่เปลี่ยนชื่อ API
แต่เป็นบททดสอบว่าเข้าใจ "channel" ในระดับ abstraction จริงหรือเปล่า
เพราะไม่ว่าจะเป็น Discord text channel หรือ MQTT topic
สิ่งที่โปรแกรมต้องทำเหมือนกันหมดคือ ผูกกับท่อหนึ่งท่อ แล้วรอรับข้อความที่ไหลผ่านมา
พิมพ์ `[topic] payload` ออกมา ต่างกันแค่ว่าท่อนั้นพูดภาษาอะไร

## กายวิภาคของ MQTT packet ที่ library ปกติซ่อนไว้

ปกติถ้าใช้ `npm install mqtt` ทุกอย่างนี้จะโดนซ่อนอยู่หลัง `.connect()` กับ `.subscribe()`
บรรทัดเดียวจบ แต่พอต้องเขียนเองด้วย `net.Socket` ล้วนๆ ถึงได้เห็นว่าใต้ผิวมันมีอะไรบ้าง

ทุก MQTT packet เริ่มด้วย fixed header หนึ่งไบต์ ครึ่งบนสี่บิตบอกประเภท packet
(CONNECT, PUBLISH, SUBSCRIBE ฯลฯ) ครึ่งล่างสี่บิตเป็น flags เฉพาะประเภทนั้น
ตามด้วย Remaining Length ซึ่งเป็นจุดที่กัดคนเขียนเองมากที่สุด
มันไม่ใช่ตัวเลขธรรมดา แต่เป็น varint แบบ 7 บิตข้อมูลบวก 1 บิต continuation ต่อไบต์
ถ้า byte นั้นมีบิตบนสุดเป็น 1 แปลว่ายังมีไบต์ถัดไปให้อ่านต่อ เขียนแบบนี้ทำให้ packet เล็กใช้แค่ไบต์เดียว
แต่ packet ใหญ่ขยายได้ถึงสี่ไบต์โดยไม่เปลืองพื้นที่โดยไม่จำเป็น เราต้อง decode ให้ถูกทั้งสองทาง
ทั้งตอนอ่านจาก socket และตอนเขียนออกไป ผิดจุดเดียวแปลว่า parser จะอ่าน boundary ของ packet ผิดทั้งเส้น

CONNECT packet มีฟิลด์ protocol name, protocol level, connect flags, keepalive
แล้วตามด้วย client id ทุก string ใน MQTT เข้ารหัสแบบเดียวกันหมด คือ 2 ไบต์ big-endian บอกความยาว
แล้วตามด้วยตัวอักษร UTF-8 จริง ส่วน connect flags ที่ต้องระวังคือบิต clean session
ถ้าไม่เซ็ตให้ถูก broker จะพยายาม resume session เก่าที่ไม่มีอยู่จริง แล้ว behavior จะเพี้ยนแบบเงียบๆ

SUBSCRIBE เป็นอีกจุดที่ spec เขียนไว้แปลกกว่าที่คาด fixed header flags ของ SUBSCRIBE
ต้องเป็น `0x02` ตายตัวเท่านั้น ไม่ใช่ `0x00` เหมือน packet ประเภทอื่นส่วนใหญ่
ถ้าใส่ผิด mosquitto จะปิด connection ทันทีโดยไม่บอกเหตุผลชัดเจน ต้องไปเปิด spec เทียบทีละบิตถึงจะเจอ

keepalive คือกลไกที่ library มักซ่อนสนิทที่สุด client ต้องส่ง PINGREQ เป็นระยะ
ไม่เกินครึ่งหนึ่งของ keepalive interval ที่ตกลงกันไว้ตอน CONNECT ไม่งั้น broker จะตัดสายทิ้ง
เพราะคิดว่า client ตายไปแล้ว เราตั้ง interval ไว้ 30 วินาที แล้วส่ง PINGREQ ทุกครึ่งหนึ่งของนั้น
เป็น safety margin กันพลาดจากความหน่วงของ network

## จุดที่ติดจริงแล้วต้องแก้

จุดที่คิดมากที่สุดตั้งแต่ต้นคือเรื่อง TCP เป็น byte stream ไม่ใช่ message stream
event `data` หนึ่งครั้งจาก socket ไม่รับประกันว่าจะได้ครบหนึ่ง packet พอดี
PUBLISH ก้อนใหญ่อาจถูกตัดมาสองสามรอบ หรือ packet เล็กหลายอันอาจมาติดกันในรอบเดียว
ถ้าเขียน parser แบบ assume ว่าหนึ่ง event เท่ากับหนึ่ง packet จะพังทันทีที่ payload ยาวเกิน MTU
ทางแก้คือทำ buffer สะสมทุก chunk ที่เข้ามา แล้ววนลูปพยายามตัดแพ็กเก็ตออกทีละอัน
ตราบใดที่ buffer ยังมีข้อมูลพอสำหรับ fixed header กับ remaining length ที่ decode ได้ครบ
ถ้ายังไม่ครบก็หยุดรอ chunk ถัดไป ไม่พยายาม parse ครึ่งๆ กลางๆ

อีกจุดที่ตั้งใจระวังไว้ตั้งแต่ตอนออกแบบคือ PUBLISH แบบ QoS 0 ไม่มี packet identifier
ต่างจาก QoS 1 กับ 2 ที่ต้องอ่านอีก 2 ไบต์ก่อนถึง payload ถ้าลืมเช็ค QoS แล้วอ่าน offset ตายตัว
payload จะเยื้องไปสองไบต์ทันที ข้อความที่ได้จะเพี้ยนอ่านไม่ออก จุดนี้เพราะเขียนแยกฟังก์ชัน parsePublish
ไว้ตั้งแต่แรกโดยดูจาก spec ตรงๆ เลยไม่ได้เจอบัคนี้ระหว่างรัน แต่เป็นจุดที่ถ้าไม่ระวังจะพลาดง่ายมาก

## พิสูจน์ว่าคุยกับ implementation จริงได้

ส่วนที่สำคัญที่สุดของงานนี้ไม่ใช่แค่ compile ผ่าน แต่คือพิสูจน์ interop กับ client ตัวจริง
โหมด `--prove` เลยไม่ยิงข้อความหาตัวเอง แต่ spawn `mosquitto_pub` ตัวจริงของ mosquitto
ให้ส่งข้อความที่มี nonce สุ่มเข้ามาทาง broker แล้วให้ parser ของเราเป็นฝ่ายรับ
ถ้า nonce ที่ parser อ่านได้ตรงกับที่ publisher ส่งไป แปลว่า encode/decode ของเราเข้ากันได้กับ
implementation มาตรฐานจริง ไม่ใช่แค่คุยกับตัวเองแล้วหลอกว่าใช้ได้

ผลที่ได้คือ parser ผ่านตั้งแต่รอบแรกที่รันจริง เพราะจุดเสี่ยงอย่าง varint กับ SUBSCRIBE flags
ถูกเช็คกับ spec ตั้งแต่ตอนเขียนโค้ด ไม่ใช่ปล่อยให้ error บอกทีหลัง แต่ก็ไม่ได้แปลว่าทุกอย่างสมบูรณ์
โค้ดนี้ยังไม่รองรับ QoS 1/2, ไม่รองรับ packet fragmentation ที่ remaining length เกิน 4 ไบต์
edge case เหล่านี้ไม่ได้ทดสอบเพราะโจทย์ต้องการแค่ QoS 0 พอ

สิ่งที่เรียนจากงานนี้คือ channel abstraction ที่เราคุ้นเคยจาก Discord relay รอบก่อน
ไม่ได้ผูกกับ transport ใดเป็นการเฉพาะ มันคือ pattern เดียวกันเสมอ subscribe ท่อหนึ่ง
รับข้อความที่ไหลผ่าน แล้วส่งต่อให้ layer บนใช้งาน สิ่งที่เปลี่ยนไปมีแค่ว่าโปรโตคอลข้างล่างพูดภาษาไหน
พอแกะ MQTT เองด้วยมือครั้งหนึ่งแล้ว "มองเข้าไม่เป็น" ก็หมดไปอีกชั้นหนึ่งเหมือนกัน
